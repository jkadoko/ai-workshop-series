# Capstone: Automated Design Verification

Now that your "operating room" is set up with the Antigravity CLI, it is time to tackle a real-world medical device scenario: **Automated Design Verification**.

In this exercise, we are going to use the AI to act as an automated test engineer. The goal is to take a list of product requirements from an Excel/CSV sheet, verify them against your product's knowledge base in NotebookLM, and automatically generate a formal Word document containing the justification for each requirement.

To do this, the AI needs to use a browser automation tool called "Playwright"—think of it as a digital robotic arm that can open web pages, click buttons, and read text for you.

### Phase 1: Understand the Workflow

Before touching the keyboard, let's look at the pipeline we are trying to build.

* **Input:** An Excel/CSV sheet containing medical device requirements (e.g., "The device must operate continuously for 4 hours on battery power").
* **Action:** The AI uses Playwright to open NotebookLM, searches your product's design specifications, and retrieves the evidence that proves the requirement is met.
* **Output:** A formatted Microsoft Word (.docx) file for *each* requirement, detailing the justification and tracing it back to the knowledge base.

### Phase 2: The "Blind" Attempt (Your Challenge)

**The Goal:** Try to get the Antigravity CLI to accomplish this task using only your natural intuition.
**The Instructions:**

1. Open your Antigravity CLI by typing `agy` in PowerShell.
2. Write a prompt asking the AI to read your provided Excel requirements sheet, use Playwright to check NotebookLM, and write the Word documents.
3. Press **Enter** and watch what happens.

*Hint: You will likely find that the AI struggles. It might give you high-level advice, write some disjointed code, or get confused about how to format the Word document. Why? Because while the AI is incredibly smart, it does not know your company's specific Standard Operating Procedures (SOPs).*

### Phase 3: The "Aha!" Moment – Introducing In-Context Training

In the medical device industry, you wouldn't tell a new engineer to "just go verify the design" without giving them a Work Instruction or an SOP. The AI is exactly the same.

To bridge this gap, we use **In-Context Training**.

Instead of expecting the AI to guess how to use Playwright or format a Word document, we provide it with a "Skill" file (saved as a `SKILL.md` file under `.agent/skills/`). A skill is essentially a digital SOP for the AI. It contains the exact, step-by-step instructions, code snippets, and formatting rules the AI needs to flawlessly execute a complex task.

### Phase 4: The Guided Attempt (Using the Skill)

**The Goal:** Achieve the deliverable by providing the AI with its SOP.
**The Instructions:**

1. In your project folder, you will notice several folders under `.agent/skills/` containing `SKILL.md` files (e.g., one for Playwright automation, one for Word document generation). **Do not open them yet!** We want the AI to read them, not you.
2. Go back to your Antigravity CLI (`agy`).
3. Write a new prompt, but this time, explicitly tell the AI to read and follow the instructions in those skill files.
* *Example prompt: "Using the requirements in [Excel File Name], perform design verification against NotebookLM. Strictly follow the procedures outlined in the provided skill files under .agent/skills/ for browser automation and document generation."*
4. Press **Enter**.

### Phase 5: Review & Approve (Human-in-the-Loop)

This is the critical governance step that separates a demo from a regulated-industry workflow.

**The Goal:** Apply the HITL review-approve-release pattern you learned in the Governance module.

**The Instructions:**

1. Open the newly generated Word documents in your editor.
2. For EACH document, perform a substantive review:
   - ✅ Is the justification technically accurate?
   - ✅ Does it correctly reference the knowledge base source?
   - ✅ Are there any hallucinated facts or incorrect citations?
   - ✅ Would this stand up to a regulatory audit?
3. Add a review header to each document (or a cover sheet):
   ```
   Generated by: Antigravity CLI + design verification skill
   Generated date: [today's date]
   Reviewed by: [Your Name]
   Review date: [today's date]
   Review status: APPROVED / APPROVED WITH MODIFICATIONS / REJECTED
   Modifications: [describe any changes made]
   ```
4. If you made corrections, document what you changed and why.
5. Commit the approved documents to Git with a descriptive message:
   ```powershell
   git add .
   git commit -m "APPROVED: Design verification reports for REQ-001 through REQ-005. Reviewed by [Your Name]."
   ```

### Phase 6: Review the Complete Audit Trail

View your project's complete history:

```powershell
git log --oneline
```

You should see a chronological record that tells the story:
1. Project setup and environment configuration
2. Skills created and tested
3. Draft documents generated by AI
4. Reviewed and approved documents committed with sign-off

This is exactly the kind of traceable, audit-ready record that regulators expect.

## Reflection: What You Accomplished

Watch as the AI transforms from a confused assistant into a precise automation tool. By feeding the AI its instructions "in context" (via the skill files under `.agent/skills/`), it now knows exactly how to drive the browser, query the knowledge base, and generate your compliance documentation.

But more importantly, you didn't just generate documents—you built a **governed workflow** with:

| Element | How You Implemented It |
|---------|------------------------|
| **Reproducibility** | Virtual environment + requirements.txt |
| **Traceability** | Git commit history with descriptive messages |
| **Standardization** | Skills (SOPs for AI) define the procedure |
| **Human Oversight** | HITL review before approval |
| **Audit Trail** | Git log + review metadata in documents |
| **Security** | Secrets in `.env`, not in code |

This is the difference between "using AI" and "implementing AI responsibly in a regulated industry."
